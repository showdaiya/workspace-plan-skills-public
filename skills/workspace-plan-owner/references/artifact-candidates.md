# artifact candidates

## 目的

workspace 内で作成・更新する成果物の候補をまとめる reference です。

成果物候補は固定テンプレートではありません。親は、plan 作成時、作業中、子への委任前、子報告の統合時にこの候補を参照し、次のセッションや委任で参照価値があるものを正式成果物として積極的に切り出します。

親は管理者として、どの成果物を残すか、どのファイル名にするか、子にどこまで作成・更新を許可するかを決めます。成果物本文の作成・更新は、原則として子へ委任します。

## 使う場面

- 初期 plan 作成時に、どの成果物を作るか検討するとき
- 作業中に、`plan.md` や `notes.md` ではなく正式成果物へ切り出すか迷うとき
- 子への委任前に、作成・更新を許可する成果物名を決めるとき
- 子の報告を受けて、報告内容を正式成果物化するか判断するとき
- `plan.md` の主要成果物一覧へ載せる候補を確認するとき

## 基本方針

- 毎回すべての候補を作らない
- 空ファイルや形だけの成果物を作らない
- 一時メモや再開ヒントだけなら `notes.md` または子の報告内の workspace メモ候補として扱う
- 要件、設計、正式な決定、方針、判断理由、調査結果、検証結果、後続 Todo の補助情報の本文は `notes.md` ではなく正式成果物へ切り出す
- 子に作成・更新させる場合は、親が委任入力でファイル名と許可範囲を明示する
- 作成した主要成果物は、必要に応じて `plan.md` の主要成果物一覧へ記録する

## 新規作成するタイミング

- 初期 plan 作成時点で必要性が明確なとき
- 一時メモでは足りず、正式な参照先が必要になったとき
- 次のセッションで読み返す判断根拠になるとき
- 子への委任で、要約だけでは不足する参照先が必要なとき
- 後続 Todo や複数の子エージェントに共通する方針・判断・補助情報があるとき
- 子の報告内容を、今後の正本として残したいと親が判断したとき

## 更新するタイミング

- 要件、方針、判断、調査結果、検証結果が変わったとき
- その成果物を後続作業の正本として最新化する必要があるとき
- 子の作業結果を、親が確認して正式成果物へ反映するとき
- rename / move / delete などで `plan.md` の主要成果物一覧との整合が必要になったとき

## 成果物候補

| file name | description |
| --- | --- |
| `requirements.md` | ユーザーが理想とする状態、必須要件、任意要件、対象外、受入条件を書く |
| `design-decision-record.md` | 要件確定までの設計意図、案の変遷、採用・不採用理由、重要な判断過程を書く |
| `decision-log.md` | 決定事項、判断理由、却下した案の要点を書く |
| `delegation-rules.md` | 子へ委任する時の範囲、禁止事項、報告条件を書く |
| `open-issues.md` | 未解決事項、確認待ち、blocked の理由と再開条件を書く |
| `change-log.md` | workspace 内の大きな変更履歴と変更理由を書く |
| `comparison-table.md` | 複数案の比較軸、比較結果、採用案を書く |
| `research-notes.md` | 調査した内容、参照元、判断に使った要点を書く |
| `implementation-plan.md` | 実装方針、変更対象、実装順序、注意点を書く |
| `test-scope.md` | テスト観点、確認手順、未検証事項を書く |
| `delegation-input-template.md` | 子へ渡す委任入力の雛形と今回固有の差し替え箇所を書く |
| `flow-validation.md` | 作業フロー、再開フロー、委任フローの検証結果を書く |
| `design-review-notes.md` | 設計不備、リスク、修正候補、レビュー結果を書く |
| `skill-draft.md` | skill 化前の下書きや構成案を書く |

## 注意

- `requirements.md` は新規 workspace の最小構成として親が初期作成する。要件のみを置き、設計意図や判断過程は必要に応じて `design-decision-record.md` / `decision-log.md` へ分離する
- `skill-draft.md` を作っても、実際の `.opencode/skills` へ配置するかどうかは別判断にする
- 成果物候補は、`plan.md` の主要成果物一覧へ全件載せるものではない
- `plan.md` の主要成果物一覧には、実際に作成した主要成果物、または作成が確定して追跡したい `planned` 成果物だけを載せる
