# workspace notes policy

## 目的

作業中に見つかった未整理メモ、再開ヒント、memory 候補（未保存）を、workspace 内 `notes.md` に一時的に残すための policy です。`notes.md` は正式成果物の代替ではなく、memory への保存も自動では行いません。

## 使う場面

- 子の報告にある `workspace メモ候補` を、どこへ残すか迷うとき
- `notes.md` に残す内容と、正式成果物へ昇格する内容の境界を確認したいとき
- 再開ヒントや未整理メモだけを workspace に残したいとき
- 長期価値がありそうな内容を、まだ memory に保存せず候補として扱いたいとき

## 基本方針

- 親は子の報告を内容別に仕分ける。進行管理は `plan.md`、要件・設計・方針・決定・判断理由・調査・検証・後続 Todo の補助情報などの正式な成果内容は workspace 成果物、未整理の一時メモと再開ヒントだけを `notes.md` に残す
- `notes.md` は非正本の補助ファイルとして扱う
- 要件定義、基本設計、正式な方針、決定ログ、判断理由、調査結果や検証結果、後続 Todo の補助情報の本体を `notes.md` に置かない
- memory へは勝手に保存しない
- 長期価値がありそうでも、まずは `memory 候補（未保存）` として扱う
- `notes.md` は session 跨ぎの再開支援を目的とし、長期知識ベースの代替にはしない
- 子エージェントのタスク終了後は、`notes.md` を再開導線と短い補足だけに絞り、膨らませない

## 3 つの語の役割

- `notes.md`: 親が仕分け後に残す、workspace 内の非正本ファイル
- `workspace メモ候補`: 子が最終報告で返す整理前ラベル。保存先そのものではない
- `memory 候補（未保存）`: 長期価値がありそうだが、まだ memory に保存していない候補

## `notes.md` の推奨構成

```md
# Notes

## 1. 目的と前提

## 2. 再開用サマリ
- 次に読むべき成果物
- 次に始める作業

## 3. 一時メモ
- まだ正式成果物に昇格していない補足
- 作業中の短い気づき

## 4. 未解決・保留
- 足りない情報
- 次回確認事項

## 5. 再開時の最初の一手
- 次に何から始めるか

## 6. memory 候補（未保存）
- 長期価値がありそうな知識
- 保存は未実施
```

正式な内容は workspace 成果物へ置き、`notes.md` にはそれらへのリンクや再開用の短い案内だけを残します。どの成果物に置くか迷うときは `artifact-candidates.md` を参照します。

## 子報告後の整理サイクル

子エージェントの報告に `workspace メモ候補` が含まれる場合、親は次の順で整理します。

1. 報告内容を読み、進行管理・正式成果・一時メモ・memory 候補に分ける
2. 進行状況、Todo 状態、次アクションは `plan.md` へ反映する
3. 要件、設計、方針、決定、判断理由、調査結果、検証結果、後続 Todo の補助情報は workspace 成果物へ反映する
4. 正式成果物にするほどではない再開ヒントや未整理メモだけ `notes.md` に残す
5. 長期価値がありそうな内容は `memory 候補（未保存）` として控え、ユーザーの明示指示があるまで memory topic は更新しない

親は `workspace メモ候補` を `notes.md` に丸ごと貼り付けません。`notes.md` は、仕分け後に残った一時メモの置き場です。

## workspace メモ候補

仕分け後に `notes.md` に残す候補:

- 正式成果物に昇格する前の短い補足
- 一時的な回避策、ハマりどころ
- 次回の再開ヒント
- 参照した成果物の関係
- plan に載せるほどではないが捨てると困る文脈

`notes.md` に残さず workspace 成果物へ昇格するもの:

- 要件定義
- 基本設計
- 正式な方針、決定事項と理由
- 調査結果の本文
- 検証結果の本文
- 後続 Todo や子への委任で参照する補助情報

## memory 候補（未保存）

長期的に価値がありそうでも、次はまだ `notes.md` や報告に候補として残すだけです。

- 再発しやすい失敗パターン
- 長く使う運用方針
- 複数 workspace をまたいで再利用する判断基準
- ユーザーの長期的な嗜好や制約

保存そのものは、ユーザーの明示指示後に `memory-ops` で行います。

## 境界の目安

### workspace メモに寄せる

- 今回の成果物や Todo と強く結びついている
- 次回この workspace を再開する人が分かればよい
- 長期保存するかまだ判断できない
- 正式な要件・設計・決定として扱うには未整理である

### workspace 成果物に寄せる

- 後続 Todo や子エージェントが正式に参照する
- 要件、設計、方針、決定、判断理由、調査結果、検証結果、タスク補助情報として残す
- `plan.md` から主要成果物として案内したい

### memory 候補にする

- 今回だけでなく今後も再利用しそう
- 他 workspace にも効く
- 再発防止や運用ルールとして価値がある

## owner メモ

- `notes.md` は memory より気軽に残してよいが、正式成果物より軽い一時メモに限定する
- ただし雑多な全文ログ置き場にはしない
- 子の報告統合後は、`notes.md` に残す内容が最小限かを必ず確認する
- plan は進行管理、workspace 成果物は内容の正本、notes は一時メモ、memory は長期知識と役割分担する
