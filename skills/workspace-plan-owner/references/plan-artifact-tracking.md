# plan artifact tracking

## 目的

`plan.md` に主要成果物を安定した名前で残し、次回再開時や子報告統合時に「何が成果物なのか」を迷わないようにするための reference です。

## 使う場面

- `plan.md` に主要成果物一覧を新しく置くとき
- 子の報告を受けて、主要成果物の path や状態を一覧へ反映するとき
- `requirements.md` や正式成果物を、次回再開時に `plan.md` からたどれるようにしたいとき
- rename / move / delete 後に、主要成果物一覧との整合を見直すとき

## 基本方針

- 主要成果物は `plan.md` に一覧を持つ
- 一時メモや一過性のログは、必要なものだけ workspace 側に置く
- 子が作成・更新した成果物は、親が報告を見て一覧へ統合する
- パスは安定したものを残し、再開時にたどれるようにする

## 推奨する表

`plan.md` に次のような表を置きます。

| path | 種別 | 関連Todo | 状態 | 用途 |
| --- | --- | --- | --- | --- |
| `plan.md` | plan | 全体 | active | 進行管理 |
| `requirements.md` | requirements | 全体 | active | ユーザー要件の初期正本 |
| `artifacts/design-summary.md` | document | T3 | done | 設計案の要約 |

`notes.md` は非正本の補助ファイルです。必要なら主要成果物一覧とは別の補助参照として扱い、要件・設計・決定の正本として載せません。

成果物候補の一覧や、新規作成・更新タイミングは `references/artifact-candidates.md` を参照します。この reference では、実際に追跡対象にした主要成果物の索引管理だけを扱います。

## 列の意味

- `path`: workspace からたどれる実ファイル名
- `種別`: plan / document / code / test / data / config / report など
- `関連Todo`: 主に関係する Todo。複数あれば `T2,T4` のように記す
- `状態`: `planned` / `in_progress` / `updated` / `done` / `blocked` / `reference` など
- `用途`: そのファイルをなぜ残すか

## 何を表に載せるか

載せるもの:

- 再開時に参照価値がある成果物
- 複数子の報告統合で見失いやすい成果物
- 設計、方針、判断、調査、検証、タスク補助情報として後続 Todo や委任で参照する成果物
- ユーザー報告や次回作業で参照される可能性が高いもの
- `requirements.md` のように、作業開始時点から正本として扱うファイル

載せなくてよいもの:

- その場限りの一時ファイル
- 自動生成物で、別の成果物から容易に再現できるもの
- 価値が薄い軽微なログ断片

## 子の報告から親が統合するルール

子報告に成果物が含まれていたら、親は次を判断します。

1. その成果物は主要成果物か
2. 次回再開時に参照価値があるか
3. 既存の表に既に載っているか
4. `状態` を更新すべきか
5. `関連Todo` と `用途` を調整すべきか

子の報告をそのまま貼るのではなく、親が表へ要約して統合します。

## rename / move / delete があった場合

- 親が許可した変更であれば、古い path を更新し、必要なら実行ログに旧 path を残す
- 親が未許可なら、子の提案として扱い、表はまだ更新しない
- 消えた成果物は、実行ログに理由を残してから反映する

## owner メモ

- `plan.md` の成果物一覧は「全文説明」ではなく「索引」に徹する
- 正式な詳細説明は成果物本体に置き、一時メモや再開ヒントだけを `notes.md` に置く
- path を安定して残すだけで、再開しやすさがかなり上がる
