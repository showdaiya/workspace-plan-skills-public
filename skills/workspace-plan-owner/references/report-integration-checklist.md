# report integration checklist

## 目的

子の報告を受けた親が、`plan.md` / workspace 成果物 / `notes.md` / plan 内の主要成果物一覧へ抜け漏れなく統合するための checklist です。
抜け漏れがあった場合は追加対応を行なってください。

## 使う場面

- 子の報告を受けた直後に、統合前の抜け漏れを確認するとき
- 報告内容を `plan.md`、workspace 成果物、`notes.md`、memory 候補へ仕分けるとき
- 設計、方針、判断、調査、検証、タスク補助情報を正式成果物へ切り出す必要があるか確認するとき
- Todo 状態、実行ログ、主要成果物一覧の更新要否を点検するとき
- blocked や未解決事項を、親やユーザー判断へ戻す形に整理するとき

## 1. スコープ確認

- 対象 Todo は依頼どおりか
- task type は依頼と一致しているか
- 完了条件・受入条件の判定が書かれているか
- 作業範囲拡張が起きていないか

## 2. 成果物確認

- 作成・更新成果物がパス付きで報告されているか
- 許可ファイル内か
- 主要成果物か、軽微な補助成果物かを分けたか
- 設計、方針、判断、調査、検証、タスク補助情報のうち、workspace 成果物として残すべき内容がないか確認したか
- rename / move / delete の有無と許可状況を確認したか
- 新規成果物化するか迷う場合、`references/artifact-candidates.md` を参照したか

## 3. plan.md 反映

- 対象 Todo の状態を `done` / `blocked` / `pending` / `cancelled` に更新したか
- 実行ログへ要約を残したか
- 完了条件未達なら理由を書いたか
- 次アクションが必要なら残したか

## 4. plan 内の主要成果物一覧の反映

- 主要成果物なら `path / 種別 / 関連Todo / 状態 / 用途` を更新したか
- 既存行の状態更新だけでよいか、新規行追加かを判断したか
- 次回たどれる path になっているか

## 5. workspace メモ候補の仕分け

- 子が返した workspace メモ候補を確認したか
- workspace メモ候補を `notes.md` にそのまま貼っていないか
- 進行状況、Todo 状態、次アクションは `plan.md` へ寄せたか
- 要件、設計、方針、正式な決定、判断理由、調査結果、検証結果、後続 Todo の補助情報は workspace 成果物へ寄せたか
- 再開ヒント、保留、未整理の補足だけを `notes.md` に残したか
- `notes.md` が再開導線と短い補足だけの最小限の状態になっているか
- `notes.md` が正式成果物の代替になっていないか
- 長期価値がありそうでも memory へ勝手に保存していないか

## 6. blocked / 未解決事項

- blocked 理由、不足情報、再開条件がそろっているか
- 未検証範囲を plan や報告に残したか
- 親またはユーザー判断が必要な点を明示したか

## 7. memory 候補の扱い

- memory 候補は候補としてだけ扱ったか
- 実保存が未実施であることを明確にしたか
- 将来 `memory-ops` に渡せる程度に理由を短く残したか

## 8. workspace 状態判断

- この workspace は引き続き `active` か
- 長期待ちで `backlog` 相当か
- 完了済みとして `archive` に閉じられるか

## owner メモ

- 子の報告をそのまま貼るのではなく、親が plan に合わせた粒度へ要約する
- 親が本文を抱え込まず、必要なら要約統合や正式成果物化も子へ委任する
- 重要なのは「次回見たときに迷わないこと」
