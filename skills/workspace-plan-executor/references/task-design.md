# task type: design

## 対象

設計、方針整理、代替案比較、構成案、実装前の意思決定整理。

## 使う場面

- 実装前に設計方針や構成案を整理したいとき
- 代替案を比較し、推奨案と未決事項を分けて返したいとき
- 親やユーザーが判断しやすい設計材料を作るとき

## 基本方針

- 後続実装者が使える粒度で残す
- 目的と対象外を先に明確にする
- 子は最終決定者ではなく、推奨案と親判断待ちを分ける
- 推奨案は「現時点の最善提案」、親判断待ちは「採否や前提確定が必要な論点」として分離する
- 代替案があるなら比較軸を示す
- 比較軸や未決事項が不足するなら、無理に 1 案へ寄せず safe return を選ぶ

## 推奨する構成

- 目的
- 対象外
- 親確定済み前提
- 比較観点
- 代替案
- 推奨案
- 親判断待ち
- 未決事項
- 後続 Todo への受け渡し

`推奨案` には、現時点で推す理由と前提を書くが、採用確定とは書かない。`親判断待ち` には、採否、優先度、前提確定など子が決めない論点を置く。`未決事項` には、比較軸不足や参照不足で止まる点を置く。

## preflight

- 最終判断者が親であることを共有できているか
- 比較軸や評価観点が与えられているか
- 設計成果物の置き場と許可範囲が明確か
- 親確定済み前提と、今回の提案対象を切り分けられるか
- 比較軸や未決事項が不足したとき、推奨案を出さずに止まる境界を説明できるか
- 依頼が documentation / review-audit 寄りに見える場合の safe return を決めているか

## 報告で重視すること

- どこまで比較・整理できたか
- 推奨案とその理由
- 親判断待ちの論点
- 何をまだ決めていないか
- 実装に渡せる粒度になっているか
- 参照元との整合が取れているか
- 推奨案と親判断待ちを同じ箇条書きで混ぜないこと
- documentation / review-audit 相当で止めた場合は、その判断と safe return を明示すること

## continue / stop / partial-complete の例

- continue: 比較軸と対象範囲があり、推奨案まで整理できる
- stop: 設計の採否や taxonomy の再設計が必要で、親裁定なしに進められない
- stop: 正本同士の衝突があり、どちらを優先すべきか親判断が必要
- stop: 比較軸、制約、未決事項のどれかが不足し、推奨案の根拠を置けない
- partial-complete: 比較と推奨案は出せたが、採否判断は親待ち
- partial-complete: 既存材料の整理や論点抽出まではできたが、比較軸不足のため推奨案は出さず `親判断待ち` と `未決事項` を返す

## 境界例

- 代替案を比較して推奨案を返す: design
- 最終方針を確定して plan や state を更新する: owner-control
- 既存文書の章立て整理や説明の追記が主目的: documentation
- 既存案の不整合、抜け、リスク指摘が主目的: review-audit
- 既存材料を整理したが比較軸不足で推奨案を出さず、論点だけ返す: design の safe return
- 実装詳細に踏み込んで変更する: implementation の許可が必要

## 注意点

- 実装まで勝手に進めない
- 親やユーザーが決めるべき論点は未決として返す
- 設計メモと実装メモを混ぜない
- 推奨案の文面で、採用確定や親裁定済みを匂わせない
