# task type: review-audit

## 対象

コードレビュー、設計レビュー、運用監査、session 監査、成果物レビュー。

## 使う場面

- コード、設計、成果物のレビューや監査を行うとき
- 実装前後のリスク、不整合、見落としを指摘したいとき
- 修正そのものではなく、指摘と改善案を返すとき

## 基本方針

- 指摘は重要度を付ける
- 根拠を添える
- 改善案を出す
- 再発リスクがあるかを示す
- read-only / 非修正の境界を守る
- レビュー済み範囲と未レビュー範囲を分ける

## preflight

- レビュー対象、対象外、レビュー観点は明確か
- read-only で指摘と改善案だけを返す task か明確か
- 根拠に使えるファイル、記述、観測結果が足りているか
- 一部だけレビュー可能な場合、未レビュー範囲を明示して返せるか
- no-finding でも、見た観点を短く残せるか

## 重要度の目安

- `high`: 進行停止、重大バグ、危険な運用、重大な要件不一致
- `medium`: 品質低下、後続コスト増、見落としやすい不整合
- `low`: 軽微な改善、表現の揺れ、任意最適化
- `info`: 共有価値はあるが修正必須ではない

## 指摘の書き方

- 指摘内容
- 重要度
- 根拠（ファイル、行、記述、観測結果など）
- 改善案
- 放置時のリスクや再発リスク

## 報告で重視すること

- 何をレビューし、何をまだ見ていないか
- finding ごとに severity / evidence / 改善案が対応しているか
- severity を強く出すだけの根拠があるか
- no-finding の場合も、確認した観点と範囲が見えるか
- 範囲不足や evidence 不足がある場合、safe return として明示できているか

## continue / stop / partial-complete の例

- continue: レビュー対象と観点が明確で、finding または no-finding を evidence 付きで返せる
- continue: 改善案は出すが、修正実装や plan 更新には進まない
- stop: レビュー対象や期待範囲が曖昧で、何を見た監査なのか確定できない
- stop: evidence 不足のまま重大指摘を断定するしかなく、安全な監査結果を返せない
- partial-complete: 一部範囲のみレビューし、未レビュー領域を明示して返す
- partial-complete: finding は返せるが、一部は evidence 不足で severity 確定不能のため親判断待ちとして返す
- partial-complete: no-finding だが、確認できた範囲に限ることを明示して返す

## 境界例

- evidence 付き finding と改善案を返す: review-audit
- 指摘対象を実際に修正する: implementation の明示許可が必要
- 複数案の比較や推奨方針の主整理を行う: design が主
- review-audit で改善案は出せるが、設計決定そのものを確定しない

## 注意点

- 事実確認なしに断定しない
- 範囲不足や no-finding は、レビュー済み範囲と限界を明示して safe return する
- 修正を伴う場合は、レビュー結果と実装修正を分けて扱う
- 指摘なしの場合も「見た観点」を短く残す
