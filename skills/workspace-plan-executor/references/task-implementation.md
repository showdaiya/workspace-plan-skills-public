# task type: implementation

## 対象

実装、修正、リファクタリング、テスト作成、既存コードの改善。

## 使う場面

- 許可されたファイルに実装や修正を入れるとき
- 実装に伴ってテスト作成や既存コード改善を行うとき
- 編集範囲や検証方法を守りつつ、動く変更を返すとき

## 基本方針

- 許可ファイルだけ編集する
- 実装本体を成立させる mechanically required な追随差分と、optional な spillover を分ける
- 実装成果物と `workspace メモ候補` を分離する
- 指定されたテスト・確認コマンドを記録する
- 主成果物は `source` として扱い、`generated` / `evidence` は補助として分けて報告する
- `generated` は実装や許可済み確認の直接結果だけを扱い、想定外 byproduct は独断で正式成果物化しない
- `temp script` は親が明示許可した場合だけ使う
- 未検証変更や追加変更を、完了扱いで包まない
- 破壊的変更が必要なら停止して親へ確認する

## preflight

- 編集してよいファイルは明確か
- 触ってはいけないファイルは明確か
- テスト・確認コマンドがあるか
- generated byproduct の扱いが明確か
- artifact category の指定有無が分かるか
- `temp script` 許可有無が明確か
- delete / rename / move の明示許可があるか
- mechanically required な追随差分として扱える範囲と、optional spillover として止める範囲が明確か

## 報告で重視すること

- どのファイルをどう更新したか
- どのコマンドを実行したか
- 成功 / 失敗 / 未実施の理由
- 受入条件と `evidence` の対応
- mechanically required な追随差分と、見送った optional spillover
- `generated` / byproduct / `temp script` の扱いと、その理由
- 未検証範囲
- `workspace メモ候補` として返すべき実装メモ

## continue / stop / partial-complete の例

- continue: 変更が許可ファイル内で閉じ、検証方法も与えられている
- continue: 追随差分が実装成立に mechanically required な範囲に留まり、追加 cleanup や standalone verification へ広がっていない
- stop: 許可外ファイルの編集が必要になった
- stop: delete / rename / move が必要になったが許可がない
- stop: 想定外の byproduct が出て、保持 / 削除 / 追記の判断に親確認が必要
- stop: mechanically required を超える追加変更がないと完了と言えなくなった
- partial-complete: `source` 更新は完了したが、一部検証が未実施、または generated artifact の扱いが親判断待ち
- partial-complete: mechanically required な追随差分までは完了したが、optional spillover は見送りまたは親判断待ち

## 境界例

- 最小の追随テスト更新が同じ許可範囲に含まれる: implementation
- 実装に直接必要な import / 呼び出し元 / 最小テスト追随: implementation
- formatter / codegen / build で出た差分は、実装成立に必要なものと想定外 byproduct を分ける
- 追加の広い cleanup や plan 更新まで進める: 範囲外
- 「ついでに」doc・verification・cleanup を広げる: optional spillover であり、完了条件に含めない
- 一時スクリプトで補助したい: 親の明示許可がある場合だけ

## 停止すべき場面

- 許可外ファイルの編集が必要になった
- delete / rename / move が必要になったが許可がない
- 外部影響や不可逆変更が必要になった
- 想定外 byproduct の保持 / 削除 / 追記判断が独断では決められない
- required verification が残っており、未検証変更を complete として返すしかなくなる
- 受入条件を満たせる見込みがなくなった

## 分離ルール

- 実装内容は対象ファイルへ
- 判断理由、再開ヒント、検証メモは workspace の成果物や `workspace メモ候補` として報告へ
