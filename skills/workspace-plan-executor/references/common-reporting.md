# common reporting

## 目的

全 task type 共通で、親が受け取りやすい報告パケットを返すための reference です。
ここで揃えるのは子の報告形式だけであり、`plan.md` への統合手順や状態更新の正本ではありません。

## 使う場面

- task type を問わず、子の最終報告の章立てをそろえたいとき
- 作成・更新した成果物、追加参照、検証結果、未解決事項を漏れなく返したいとき
- blocked や途中完了でも、親が次の判断をしやすい報告にしたいとき

## 共通ルール

- 報告は「実施範囲」「結果」「未解決」を分ける
- 主 `task type` を報告の主軸にし、`extra task types` は補助観点や追加報告項目として添える
- 作成・更新した成果物は、必ずパス付きで返す
- 追加参照したファイルもパス付きで返し、必要になった理由を短く添える
- 親から渡された Memory Context 要約が今回の判断や範囲にどう影響したかを返す。影響がなければ「影響なし」と明記してよい
- Memory Context を使った場合も、memory 詳細調査を自分で広げていないかを短く返す
- 編集対象が限定されるときや並行委任では、触ったファイルと触らなかったファイルを分けて返す
- workspace メモ候補を返す。`notes.md` へ直接書くのではなく、どこへ残すかは親が判断する
- Todo 状態や workspace 状態は子が決めず、親が判断できる材料として返す
- `plan 反映候補` は親が統合判断するための入力であり、子が plan 更新手順を定義するものではない
- memory へは保存しない。必要なら候補として返すだけにする
- task-specific な stop trigger に当たった場合は、trigger の内容と、そこから導いた `完了 / partial / blocked / not-run` を共通 packet 上で対応づけて返す

## 最低限含める章

- 実施した Todo 範囲
- 結果サマリ（完了 / 部分完了 / blocked のどれに近いか）
- 作成・更新した成果物
- 触ったファイル / 触らなかったファイル
- 受入条件マトリクス
- artifact category 別報告
- 参照したファイル
- 追加参照した理由
- Memory Context の扱い
- 検証結果
- 親判断待ち
- plan 反映候補
- workspace メモ候補
- 親が plan に反映するための状態判断材料
- 削除・リネーム・移動の有無と、親の許可状況
- 未解決・注意点
- memory へ移す候補（あれば）

blocked / partial / 親確認返しでは、次も共通 packet のラベルとして優先します。

- 不足情報
- 再開条件
- 実施済み / 未完了
- 未検証範囲
- 親へ確認すべき内容
- 推奨対応

重くしすぎないため、`evidence` / `temp script` / `memory 候補` など存在しない章は「なし」と明記するか、省略可とします。

## 受入条件マトリクス

受入条件ごとに、最低限次の列をそろえます。

- `受入条件`
- `結果`: `pass` / `fail` / `partial` / `not-run` / `判定不能`
- `根拠 / evidence`
- `confidence`
- `補足`

簡易例:

| 受入条件 | 結果 | 根拠 / evidence | confidence | 補足 |
| --- | --- | --- | --- | --- |
| SKILL.md に主 / extra の優先順位が追記されている | pass | `/agent_work/.../SKILL.md` の該当節 | high | wording adjustment のみ |

## artifact category 別報告

必要なものだけ、次の category で分けて返します。

- `source`: 主成果物。どのファイルを作成 / 更新したか
- `generated`: 許可されたコマンドやツール実行に伴う生成物。保持したか、報告だけか
- `evidence`: 受入条件や検証結果の根拠に使ったもの
- `temp script`: 親が明示許可した場合だけ。作成有無と、保持 / 削除を返す

artifact category は報告整理用であり、許可範囲を広げる根拠にはしません。

## 成果物の書き方

次の粒度を推奨します。

- `path`: 何をしたか（作成 / 更新 / 提案のみ）
- 用途または変更概要
- 必要なら artifact category

例:

- `/agent_work/workspaces/active/.../artifacts/design.md`: 更新。比較観点と推奨案を追記（source）
- `/agent_work/tmp/.../verification.log`: 生成。検証ログを evidence として参照（generated / evidence）

## 触ったファイル / 触らなかったファイル

編集可能ファイルが限定されるとき、または並行委任で衝突回避が重要なときは、少なくとも次を返します。

- `触ったファイル`: 作成 / 更新 / 参照のみを区別して列挙する
- `触らなかったファイル`: 親の制約、衝突回避、不要判断など、触らなかった理由を短く添える

すべてのファイルを網羅する必要はありません。親が今回の委任範囲と衝突有無を判断できる粒度を優先します。

## Memory Context の扱い

Memory Context が渡されている場合は、最終報告で次を短く返します。

- 親から渡された Memory Context 要約のうち、今回使った要点
- その要点が作業範囲、判断、注意点にどう影響したか
- memory 詳細調査を自分で広げていないこと。追加確認した場合は親の指定範囲かどうか
- Memory Context と現状確認が矛盾したかどうか

影響がない場合も「参照したが今回の変更判断には未使用」など、扱いを見える化します。

## plan 反映候補の書き方

親が統合しやすいよう、候補を短く分けて返します。

- `新規 Todo 候補`
- `Todo 再設計候補`
- `成果物一覧更新候補`
- `前提変更候補`

ここで返すのは候補だけです。どれを採用するか、どう plan に反映するかは親が判断します。

## workspace メモ候補の書き方

親が `notes.md` / workspace 成果物 / memory 候補へ仕分けられるよう、再開に役立つ内容を短く返します。`workspace メモ候補` は子側の報告ラベルであり、保存先そのものではありません。

- 判断理由
- ハマりどころ
- 未解決だが今すぐ memory に上げないこと
- 次回の最初の一手

## 状態判断材料の書き方

親が `plan.md` や workspace 状態を判断しやすいよう、状態名を断定せずに根拠を返します。

- 完了条件をどこまで満たしたか
- blocked なら何が不足しているか
- 続行時に親が判断したい論点があるか
- workspace 状態に影響する未解決があるか

## memory 候補の扱い

- 長期価値がありそうなものだけ候補として返す
- 直接保存しない
- 「なぜ長期価値があるか」を 1 行で添える

## blocked / 部分完了 時

- 途中までできた内容も返す
- 未検証範囲を隠さない
- 親が次に判断しやすいように、`不足情報` と `再開条件` を分けて書く
- `実施済み / 未完了` を分け、どこまで進んだかを明示する
- `親へ確認すべき内容` と `推奨対応` を添え、親判断待ちの論点を短く返す
- 部分完了なら「どこまで完了し、何が親判断待ちか」を明示する
- task reference の stop trigger に当たった場合は、「どの trigger で止まったか」→「結果分類」→「不足情報 / 再開条件」の順でつなげて返す
