# task type: documentation

## 対象

README、設計書、整理メモ、説明文、手順書、引き継ぎ用の文書成果物。

## 使う場面

- README や手順書、説明文を作成・更新するとき
- 既存実装や設定を、読み手向けに整理して文書化するとき
- 実装は変えず、説明や引き継ぎ資料を整えるとき

## 基本方針

- 読み手を意識して粒度を決める
- 参照元を明確にする
- 実装や現状と整合しているか確認する
- canonical source が衝突する場合は、子で裁定せず safe return を優先する
- `現行仕様` / `提案中` / `記載保留` を分ける
- 根拠のない断定を書かない
- 一時的な説明文で済むものと、持続する文書成果物を混同しない

## preflight

- 読み手は誰か
- どの場面で使う文書か
- 概要レベルか、手順レベルか
- 正本の優先順位は明確か
- canonical source 同士の conflict はないか
- 更新可能ファイルと、既存文書との関係は明確か
- 参照元の実装や設定と矛盾していないか

## 報告で重視すること

- 作成・更新した文書パス
- 文書の用途
- 参照元
- 実装との整合確認結果
- `現行仕様` / `提案中` / `記載保留` の分離結果
- 未確定のため記載を保留した点

## continue / stop / partial-complete の例

- continue: 読み手、正本、更新対象ファイルが明確で、現行事実に基づいて更新できる
- stop: canonical source 同士が衝突し、どの記述を優先すべきか親判断が必要。safe return として conflict 箇所を返す
- stop: 文書更新だけでは済まず、実装変更や scope 外リンク更新が必要
- partial-complete: 非衝突の `現行仕様` だけ更新し、`提案中` と `記載保留` を分けて返す

## 境界例

- 現行の確認済み仕様を文書へ反映する: documentation
- 提案中の内容を本文で確定仕様のように書く: 不可。必要なら提案中と明記する
- 実装側の修正まで含める: implementation の許可が必要
- ユーザー向けの一時的な説明文や報告文を整える: communication-summary。正本となる文書成果物更新とは分ける

## 注意点

- 不明な仕様を埋め草で書かない
- canonical conflict を文面で吸収して断定しない
- 実装変更が必要なら documentation task の範囲外として親へ返す
