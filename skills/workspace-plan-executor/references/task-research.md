# task type: research

## 対象

read-only 調査、監査、比較、既存実装確認、仕様確認、現状把握。

## 使う場面

- 既存コード、文書、設定を read-only で調査するとき
- 実装前に現状把握や比較材料を集めたいとき
- 原因調査や仕様確認として、確認できた事実を整理して返すとき

## 基本方針

- まずローカル情報を確認する
- 事実 / 推測 / 未確認を分ける
- 参照したファイルを列挙する
- 深掘りは、親が示した観点を満たした時点か、追加参照が同じ論点の反復になった時点で打ち止めにする
- read-only 調査を基本とし、generated な調査成果物は親が許可した場合だけ残す
- 読みやすさのためだけに抽出表・比較メモ・調査成果物を新規作成したくなっても、親の許可がなければ報告本文で返す
- 外部調査の可否が不明で、外部情報が必要なら停止して親へ確認する

## preflight

- 調査対象範囲と観点は明確か
- 外部調査の可否は明確か
- 許可された成果物や出力先があるか
- 深さや打ち止め条件が示されているか。未指定なら、主要観点を満たした時点で safe stop できるか
- 事実整理だけで足りるか、比較や監査観点まで必要か

## 進め方

1. 親が示した対象範囲と観点を確認する
2. `plan.md` と関連成果物を読み、調査対象を絞る
3. 必要最小限の追加参照を行う
4. 参照元同士が衝突したら、無理に一本化せず conflict 箇所を明示する
5. 事実を整理し、必要なら比較や差分をまとめる
6. 推測や未確認事項、打ち止め理由は明示する

## 報告で重視すること

- 確認できた事実
- まだ確認できていない点
- 推測が混ざる箇所
- どこで深掘りを打ち止めにしたか
- source conflict があった場合の safe stop / partial の境界
- 参照ファイル一覧
- 親が次に判断すべき点

## continue / stop / partial-complete の例

- continue: ローカル参照だけで主要観点を確認でき、許可範囲内の報告で完結する
- stop: 外部調査や許可外の新規成果物がないと前進できない
- stop: 参照元同士が衝突し、親の裁定なしに結論化できない。safe stop として conflict を明示して返す
- partial-complete: 主要事実は整理できたが、一部観点だけ未確認 source や source conflict が残る

## 境界例

- 既存コードと既存文書を read-only で比較する: research
- 調査結果を既存文書へ反映する: documentation または implementation 側の許可が必要
- 抽出表や比較メモをファイルとして残す: 親が成果物として許可した場合だけ
- 深掘りしたくなるが、主要観点を満たした後に追加参照が反復になる: safe stop して未確認だけ返す

## 注意点

- 明示許可がない限り編集しない
- 結論を急いで断定しない
- 参照元 conflict を、追加探索で必ず解消できる前提にしない
- generated な調査成果物へ逃がさず、許可がなければ本文報告と参照列挙で閉じる
- 外部情報を使う場合は、親の許可・出典条件・安全性を確認する
